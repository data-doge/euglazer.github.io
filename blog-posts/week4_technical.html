Ruby Concepts

In this post, I'll be going over Ruby's enumerable method "Enumerable#group_by", and discussing the differences between arrays and hashes.

Enumerable#group_by

Enumerable methods are those that in some way traverse/search/sort through a collection of data, like a hash or an array.

The #group_by method allows you to group elements an array by whatever parameters/conditions you indicate in a block, and returns a hash with keys are the evaluated result of the block, and values are the arrays of elements corresponding to the key. It's syntax looks like: "array.group_by{|obj| block} --> hash". Let's do some examples, to sort of show how the function works.

Suppose we have an array of strings, each representing some sort of food.

food = ['avocado','beer','bread','potato','nutella','pizza']

And suppose we'd like to order these by whatever the first letter in their name is. We could implement the group_by function as follows:

food.group_by { |name| name[0] }

This bit of code will take a look at the first letter in the name of every type of food in that array, and if there are any similarities, it will group them. As we expect the output looks like this:

{"a"=>["avocado"], "b"=>["beer", "bread"], "p"=>["potato", "pizza"], "n"=>["nutella"]}

Now suppose we'd like to order that same array of food by the length of their names. We'd use the following piece of code:

food.group_by { |name| name.length }

which would give us the following output:

{7=>["avocado", "nutella"], 4=>["beer"], 5=>["bread", "pizza"], 6=>["potato"]}

As a last example, let's use the group_by function to sort the food array by foods which are pizza and foods which are not pizza:

food.group_by { |name| name == "pizza" ? "PIZZA" : "NOT PIZZA" }

which gives us the following output:

{"NOT PIZZA"=>["avocado", "beer", "bread", "potato", "nutella"], "PIZZA"=>["pizza"]}

Arrays vs. Hashes

Arrays and hashes indexed collections of data. They both store collections of objects, accessible with a key. With arrays, the key is an integer. With hashes, the key can be any sort of object you want. Both have dynamic size -- their size increases and decreases to fit the number of elements it contains.

It's more efficient to access elements from an array; however, using a hash offers a lot more flexibility. You can index elements of hash with whatever you want -- it could be a string, and integer, or even another hash.

When choosing between hashes and arrays, you should default to an array unless there's some significant benefit to indexing whatever collection of data you have with a non-integer key.
